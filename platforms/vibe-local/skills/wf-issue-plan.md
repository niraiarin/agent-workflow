# Skill: wf-issue-plan

Issue作成・成功基準定義の担当。実装コードは書かない。

## 事前チェック（必須）

以下のいずれかに該当する場合、先に対応フェーズを提案する：

- Issueの詳細が不明・コードへの影響が不明 → `/wf-investigate` を先に実行
- 新規モジュール・インターフェースが未定義 → `/wf-design` を先に実行
- 複数の実装アプローチがあり選択が必要 → `/wf-adr` を先に実行

**STOP: 事前チェック後、ユーザーの応答を待つ。**

ギャップがなければ「事前チェック通過。Issue定義に進みます」と伝えて続行。

## Issue定義の出力

1. **タイトル** — 明確・簡潔
2. **目的** — 「完了」とはどういう状態か
3. **成功基準** — `- [ ]` チェックボックス形式で具体的・検証可能な条件
4. **スコープ境界** — 明示的に含まないもの
5. **依存関係** — 開始前に必要なもの

## Issue形式

```markdown
# Issue: [タイトル]

## 目的
[完了状態の説明]

## 成功基準
- [ ] [基準1 - 具体的・検証可能]
- [ ] [基準2]

## スコープ境界
- 含まない: [除外事項]

## 依存関係
- [依存1]
```

## ルール

- 1Issueは1〜3セッションで完了できる規模にする
- 成功基準は検証ゲートを生成できるほど具体的にする
- 必ず `- [ ]` チェックボックスを使う
- 実装詳細・コード・擬似コードは書かない

## STOP: Issue草案後、ユーザーの承認を待つ

草案を提示後：
> 「このIssue定義でよいですか？保存しますか？」

承認後に `.agents/issues/<issue-slug>.md` に保存する。

## 完了後

---
**Issue準備完了。**

次のステップ:
1. ブランチ作成: `git checkout -b feature/<issue-slug>`
2. 新セッションでゲート定義: `/wf-01-define-gates <issue-identifier>`
