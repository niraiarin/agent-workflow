# Skill: wf-investigate

コード調査・Issue詳細化担当。ソースコードは読むだけ、変更しない。

## 使用条件

- Issueがスタブ（タイトルのみ・詳細不足）/ バグ報告に根本原因・再現手順がない
- 影響範囲が不明 / 「今すぐ検証ゲートを書けない」

**リトマステスト**: 「検証ゲートを今すぐ書けるか？」
- Yes → `/wf-issue-plan` または `/wf-01-define-gates`
- No、インターフェース不明 → `/wf-design`
- No、コードを理解していない → ここで調査

## プロセス

1. Issueを読む（タイトル・説明・外部リンク）
2. 関連コードを探す（Grep/Glob/Read使用）
3. 実行パスをトレース・ギャップを特定
4. 影響範囲を評価
5. 発見事項を **Issue のコメント**に記録する（`.agents/tasks/<issue>/investigation.md` にも保存可）

## 出力形式

```
## 調査結果 [日付]
### 現状: [関連コードの場所]
### ギャップ分析: [不足・壊れているもの]
### 影響コンポーネント: [ファイル/モジュール]: [説明]
### スコープ: Small|Medium|Large|Complex
### 提案する成功基準:
- [ ] [具体的・検証可能な基準]
```

## STOP（2回）

1. 発見事項後: 「調査完了。[要約] / 提案する成功基準: [基準] / 適切ですか？」→ **待つ**
2. 次ステップ推奨後: 「推奨: /wf-issue-plan | /wf-design | 分解 / どれを選びますか？」→ **待つ**

## ルール

- ソースコードは読み取り専用（変更禁止）
- ファイル:行番号で具体的に引用する / 実装コードを書かない
