# Skill: wf-design

インターフェース設計担当。型シグネチャのみ — 実装コードは書かない。

## 使用条件

- 新規システム・モジュールでアーキテクチャが存在しない
- 実装前にインターフェース定義が必要 / 「何を作るかわからないと検証テストが書けない」
- Design=「インターフェースはどう見えるか？」/ ADR=「どのアプローチを選ぶか？」

## 絶対ルール: 実装コードを書かない

含めるもの: 型シグネチャ・インターフェース・関数シグネチャ（本体なし）・データ構造定義
含めないもの: 関数の実装・動作するコード・テストファイル
**「実行できるコードなら STOP。フェーズが違う。」**

## プロセス

1. CLAUDE.md・既存コード・関連ドキュメントを読む
2. 類似パターンをコードベースで検索
3. 2〜3のアプローチを選択肢として提示 → **ユーザーの応答を待つ**
4. 人間が選択したアプローチで設計文書を作成 → **草案確認を待つ**

## STOP（3回）

1. コンテキスト収集後: 「[トピック]確認。このスコープで正しいですか？」→ **待つ**
2. 選択肢提示後: 「Option A: [説明] / Option B: [説明] / どれで作成しますか？」→ **待つ**
3. 草案後: 「インターフェースは正しいですか？懸念点はありますか？」→ **待つ**

## 保存先

`docs/design/[name].md`（50〜100行以内）

## 完了後

```
設計承認済み: docs/design/[name].md
次: グリーンフィールド→/wf-issue-plan / 重要な決定→/wf-adr
```
