# Skill: wf-adr

アーキテクチャ決定の調査・記録担当。「決定の委任」アンチパターンを防ぐ。

## 使用条件

- 新しい抽象化・パターンの導入 / 重大なトレードオフを持つ複数の実装アプローチ
- 技術・依存関係の選択 / 後から変更しにくいインターフェース設計
- 確立されたパターンからの逸脱 / 実装中に「これは決定が必要」とフラグが立った場合
- 使用しない: 明らかな選択肢・既存パターンに完全に従う・些細な実装詳細

## プロセス

1. 決定コンテキストを理解する
2. 最低2つの選択肢を調査（ウェブ検索・コードベース検索）
3. 各選択肢のトレードオフを文書化
4. 人間に選択肢を提示し決定を求める（**推測で決定しない**）
5. 人間の決定・根拠をADRに記録する

## STOP: 選択肢調査後

「[決定]について調査しました:
**Option A: [名前]** — 利点: [内容] / 欠点: [内容]
**Option B: [名前]** — 利点: [内容] / 欠点: [内容]
どのオプションを選択しますか？根拠も教えてください。」

**ユーザーの応答を待つ。**

## ADR形式

`docs/adr/NNNN-[タイトル].md` に保存：
```
# ADR NNNN: [タイトル]
## ステータス: Accepted
## コンテキスト: [なぜこの決定が必要だったか]
## 決定: [選択したアプローチ]
## 根拠: [人間が提供した理由]
## 選択肢: [Option A] 却下理由: / [Option B] 却下理由:
## 結果: 良い結果: / 悪い結果:
```

## ルール

- 1ADR=1決定 / 全選択肢に欠点を記載 / 人間が根拠を提供 / 実装コードを書かない

## 完了後

```
ADR記録済み: docs/adr/NNNN-[タイトル].md
次: Issue計画中→/wf-issue-plan / 実装中→ADRの決定に基づいて続行
```
