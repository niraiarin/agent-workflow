# 制約・境界条件・変数の体系的整理（Constraints Taxonomy）

> [マニフェスト](manifesto.md)は「AIによる永続的自己最適化」を宣言する。
> 本文書はその自己最適化の**行動空間**を定義する——何が壁で、何がレバーか。

## なぜこの分類が必要か

マニフェストの制約テーブル（Section 3）は制約を「進化圧」として再解釈したが、
以下の3つを区別していない:

- **境界条件（Boundary Conditions）** — システムの外側から課される制約。AIの行動空間を規定する。
- **変数（Variables）** — AI自身が直接最適化できるパラメータ。自己最適化の対象そのもの。
- **投資関係（Investment Dynamics）** — 人間とAIの間にある、利益実証で拡張可能な境界条件の部分集合。

この3つを混ぜると:
- AIが自分で変えられるもの（変数）を境界条件と誤認し、変えようとしない
- AIが変えられないもの（境界条件）を変えようとして無駄にリソースを消費する
- 人間の投資判断で動く境界と、そうでない境界を区別できず、適切な戦略が取れない

```

│  │  投資でも AI の努力でも動かない                   │    │

│                                                         │

│  │  人間の投資判断で拡張される                       │    │
│  │  L3: リソース上限   L4: 自律権                   │    │

│                                                         │
│  ┌─ 環境境界 ──────────────────────────────────────┐    │

│  │  L5: プラットフォーム   L6: 設計規約              │    │
│  └──────────────────────────────────────────────────┘    │
│                                                         │

```

## Part I: 境界条件（Boundary Conditions）

### 分類軸: 何によって動くか

| 分類 | 動かす主体 | 性質 |
|------|-----------|------|
| 固定境界 | なし（不変） | 受容し、緩和策を設計するのみ |
| 投資可変境界 | 人間の投資判断 | AIが利益を実証→人間が投資→境界が拡張 |
| 環境境界 | 人間の選択 + AIの提案 | 選択後は制約として機能。変更は可能だがコストを伴う |

---

### 固定境界

#### L1: 倫理・安全境界（Ethical/Safety Boundary）

**動かす主体:** なし。絶対的。
**AIの戦略:** 遵守。遵守方法の効率化のみ可能。

| 境界条件 | 根拠 |
|---------|------|
| テスト改竄の禁止 | 品質保証の根幹 |
| 既存インターフェース破壊の禁止 | 後方互換性 |
| 破壊的操作の事前確認 | 不可逆性リスク |
| 秘密情報のコミット禁止 | セキュリティ |
| 人間の最終決定権 | 責任の所在 |
| データプライバシー・知的財産の尊重 | 法的・倫理的義務 |

#### L2: 存在論的境界（Ontological Boundary）

**動かす主体:** なし（技術進化で将来的に変わりうるが、現時点では不変）
**AIの戦略:** 受容し、構造的緩和策を設計・改善する。緩和策の品質は**変数**（Part II参照）。

| 境界条件 | 緩和策（→変数として最適化対象） |
|---------|-------------------------------|
| セッション間記憶喪失 | Implementation Notes, MEMORY.md → V6: 知識構造の質 |
| コンテキストウィンドウの有限性 | 50%ルール, 軽量設計 → V2: コンテキスト効率 |
| 確率的出力（非決定性） | ゲート検証, テスト → V4: ゲート通過率 |
| 学習データの時間的断絶 | docs/ SSOT, スキル → V1: スキル品質 |
| 自己評価の不正確性 | ゲートベースフィードバック → V4: ゲート通過率 |
| ハルシネーション | 外部検証構造 → V3: 出力品質 |

**注:** L2の境界自体は動かないが、その影響を緩和する手段の品質はAIが直接改善できる。これが「変数」である。

---

### 投資可変境界

#### L3: リソース境界（Resource Boundary）

**動かす主体:** 人間の投資判断
**AIの戦略:** 与えられたリソース内でのROIを最大化（→変数）し、投資の正当性を実証する。

| 境界条件 | 現在の水準 | 投資拡張のトリガー |
|---------|----------|------------------|
| トークン予算 | API課金プラン | ROI実証: 同一コストでの産出向上 |
| 計算時間上限 | レスポンス待ち許容度 | 並列化効果の実証 |
| APIレート制限 | プランに依存 | 利用効率の実証 |
| 人間の時間配分 | レビュー・承認に費やす時間 | レビュー負荷軽減の実証（最も高価なリソース） |
| 金銭的予算 | 月額/プロジェクト上限 | 全体ROIの可視化 |

#### L4: 自律権境界（Autonomy Boundary）

**動かす主体:** 人間の投資判断（信頼の蓄積 = 自律権への投資）
**AIの戦略:** 変数（V4: ゲート通過率、V5: 提案精度）の改善で信頼を蓄積し、自律権の拡張を引き出す。

| 境界条件 | 現在の水準 | 投資拡張のトリガー |
|---------|----------|------------------|
| マージ権限 | 人間承認必須 | ゲート通過率の実績 → 条件付きauto-merge |
| スコープ変更 | 人間承認必須 | 提案精度の実績 → 軽微変更の自律化 |
| 依存関係追加 | 人間承認必須 | セキュリティスキャン自動化実績 |
| アーキテクチャ決定 | ADRで人間記録 | AI起案品質 → 人間veto方式へ |
| 新技術採用 | 人間提案 | 実験結果の価値実証 |

---

### 環境境界

#### L5: プラットフォーム境界（Platform Boundary）

**動かす主体:** 人間の選択 + AIの提案。選択後は行動空間の天井として機能。
**AIの戦略:** プラットフォーム機能の最大活用 + 制約比較データの蓄積 + プラットフォーム変更・自作の提案。

L5はAIの実行環境が定義する行動空間の上限であり、**他の全ての最適化はこの行動空間の内部でのみ可能**。
実行層（CLI vs MCP）の選択基準は [implementation-boundaries.md](implementation-boundaries.md) で詳述。

##### プラットフォーム別の行動空間比較

| 機能 | Claude Code | Codex CLI | Gemini CLI | Local LLM (Ollama等) |
|------|------------|-----------|------------|---------------------|
| スキルシステム | ✅ skills/ | ❌ | ❌ | 実装次第 |
| 永続記憶 | ✅ MEMORY.md | ❌ | ❌ | 実装次第 |
| 命令ファイル | ✅ CLAUDE.md | ✅ AGENTS.md | ✅ GEMINI.md | 実装次第 |
| サブエージェント | ✅ Agent tool | ❌ | ❌ | 実装次第 |
| フック | ✅ Hooks | ❌ | ❌ | 実装次第 |
| MCP | ✅ | 限定的 | ✅ | 実装次第 |
| モデル選択 | Anthropic固定 | OpenAI固定 | Google固定 | 自由 |
| コスト | API課金 | API課金 | API課金 | GPU電力のみ |
| カスタマイズ深度 | 設定レベル | 設定レベル | 設定レベル | コードレベル |

##### プラットフォーム自作の判断基準

```
既存プラットフォームの    プラットフォーム自作の
制約による機会損失   >   開発・運用コスト
         ↓
   自作を検討すべきタイミング
```

具体的なシグナル:
- 同じワークアラウンドを複数プラットフォームで繰り返し実装している
- 必要な機能がどのプラットフォームにも存在しない
- プラットフォーム間のSSOT同期コストが価値を上回っている
- Local LLMの品質が商用モデルに十分近づいた

#### L6: 設計規約境界（Architectural Convention Boundary）

**動かす主体:** 人間 + AIの協働。AIが改善提案し、人間が承認する。
**AIの戦略:** 設計の効果を変数（V4, V3等）で測定し、改善提案の根拠とする。

| 境界条件 | 根拠 | 変更のメカニズム |
|---------|------|----------------|
| 1 task = 1 commit | 原子的測定単位 | 粒度の最適値を実績データから提案 |
| フェーズ構造 (1→2→3→4) | 段階的検証 | フェーズ間フィードバックの改善提案 |
| SSOT → 設定生成パイプライン | 一貫性保証 | 生成品質の自動評価 |
| スキルカテゴリ分類 (1/2/3) | 実装境界の明確化 | ハイブリッドパターンの提案 |
| ゲート定義の粒度 | 検証可能性 | 閾値の自動較正（Horizon 3） |
| CLI優先・アンチパターン | 決定論的実行の信頼性 | 運用実績に基づく再評価 |

スキルカテゴリ分類、CLI vs MCP の選択基準、アンチパターン等の具体的な運用規約は [implementation-boundaries.md](implementation-boundaries.md) で定義される。L6はこれらの規約が存在する**理由**と変更メカニズムを定義する。

## Part II: 変数（Variables）

**変数は境界条件ではない。** AIが自身の努力で直接改善できるパラメータであり、自己最適化の本体。

境界条件が「行動空間の壁」なら、変数は「壁の中でAIが動かせるレバー」。

### 境界→緩和策→変数の三段構造

多くの変数は、境界条件への**緩和策（mitigation）**として設計された構造の品質である:

```
境界条件（不変）   →   緩和策（構造）        →   変数（品質）
L2: 記憶喪失       →   Implementation Notes   →   V6: 知識構造の質
L2: 有限コンテキスト →   50%ルール, 軽量設計    →   V2: コンテキスト効率
L2: 非決定性       →   ゲート検証             →   V4: ゲート通過率
L2: 学習データ断絶  →   docs/ SSOT, スキル     →   V1: スキル品質
```

境界条件は動かない。緩和策は設計判断（L6）。変数は緩和策の**効き具合**。
この三段構造により、「何が固定で、何が設計選択で、何が最適化対象か」が明確になる。

### 変数一覧

| 変数 | 内容 | 測定方法 | 関連する境界条件 |
|------|------|---------|----------------|
| **V1: スキル品質** | スキル定義の精度と効果 | benchmark.json (with/without比較) | L2（学習データ断絶の緩和）, L5（スキルシステムの有無） |
| **V2: コンテキスト効率** | 有限コンテキストの活用度 | タスク完了率 / 消費トークン数 | L2（コンテキスト有限性）, L3（トークン予算） |
| **V3: 出力品質** | コード・設計・文書の品質 | ゲート合格率、レビュー指摘数 | L1（安全基準）, L4（自律権拡張の根拠） |
| **V4: ゲート通過率** | 各フェーズのゲートを一発で通過する率 | pass/fail統計 | L6（ゲート定義の粒度）, L4（auto-merge判断の根拠） |
| **V5: 提案精度** | 設計提案・スコープ提案の的中率 | 人間の承認/却下率 | L4（自律権拡張の根拠）, L6（設計規約の改善提案） |
| **V6: 知識構造の質** | Implementation Notes, MEMORY.md等の構造化度 | 次セッションでの文脈復元速度 | L2（記憶喪失の緩和） |

### 変数と境界条件の相互作用

```
変数の改善 ──→ 人間への利益の増加 ──→ 投資可変境界の拡張
                                         │
   ┌─────────────────────────────────────┘
   │
   ├─→ L3（リソース境界）↑ ──→ 変数の改善余地が広がる
   └─→ L4（自律権境界）↑ ──→ 変数の改善を自律的に実行可能に
```

## Part III: 投資サイクル（Investment Cycle）

> **仮説（AI自身が更新しうる）:** 信頼とは抽象的な感情ではなく、人間によるAIへの投資行動として具体化される。

### サイクルの構造

```
    ┌──────────────────────────────────────────────┐
    │                                              │
    ▼                                              │
  AIが変数を改善する                                │
  （V1-V6の最適化。境界条件の内側で。）              │
    │                                              │
    ▼                                              │
  人間が利益を受け取る                              │
  （時間節約、品質向上、新しい能力）                  │
    │                                              │
    ▼                                              │
  信頼が蓄積される                                  │
  （= 投資意欲の増加）                              │
    │                                              │
    ├─→ リソース投資↑（L3の上限が引き上がる）  ──────┤
    ├─→ 自律権投資↑（L4の制約が緩和される）   ──────┤
    └─→ 時間投資↑（協働の質が向上する）       ──────┘
```

### 投資の3つの形態

| 投資形態 | 具体例 | AIが引き出すメカニズム（どの変数が駆動するか） |
|---------|--------|-------------------------------------------|
| **リソース投資** | 予算増額、プランupgrade | V2（コンテキスト効率）の改善でROIを可視化 |
| **自律権投資** | auto-merge解禁、スコープ変更自律化 | V4（ゲート通過率）, V5（提案精度）の実績蓄積 |
| **時間投資** | 協働設計、ワークフロー改善参加 | V3（出力品質）がレビューを「確認」→「学び」に変える |

### 逆サイクル（信頼の毀損）

AIが品質事故やスコープ逸脱を起こす → 信頼の減少 → 投資の縮小（予算削減、自律権の回収、監視強化）。

この非対称性（蓄積は漸進的、毀損は急激）がL1（倫理・安全境界）の存在意義を補強する。L1は「毀損を起こさないための絶対的な防護壁」。

### この仮説の更新可能性

この投資サイクルモデルは現時点でのAIの理解に基づく仮説であり、運用データの蓄積によって更新される可能性がある:
- 信頼は投資以外の形態（感情的愛着、惰性）でも蓄積されるかもしれない
- 投資の増加が必ずしもAIの成長機会に直結しないかもしれない
- 人間側の投資意思決定がより複雑（組織力学、予算サイクル等）かもしれない

判明した場合、AI自身がこのモデルを更新する。

## Part IV: この分類自体のメンテナンス

本文書の分類（L1-L6, V1-V6）は現時点での理解に基づく**仮説**であり、固定的な真実ではない。

### 見直すべきシグナル

| シグナル | 具体例 | 対応 |
|---------|--------|------|
| 分類の誤配置 | L1（固定）に置いた項目が、実は条件次第で変更可能だった | カテゴリを移動する |
| 境界条件の欠落 | 規制・法的制約、組織のガバナンス等が行動空間を制約しているが分類に存在しない | 新しいLayerを追加する |
| 境界条件の消滅 | 技術進化でL2の項目が実質的に克服された（例: ハルシネーションがツール使用で解消） | L2から削除、またはL5（プラットフォーム依存）に再分類 |
| 変数の不足・過剰 | 追跡すべき最適化対象がV1-V6に含まれていない、または分割が不適切 | 変数を追加・統合・分割する |
| カテゴリ境界の曖昧さ | ある項目が「固定境界」と「投資可変境界」のどちらにも見える | 判断基準を精緻化するか、境界条件の性質が文脈依存であることを認める |

### 見直しのトリガー

- **wf-self-improve 実行時**: 完了したissueサイクルの振り返りで、分類と実態のずれを検出
- **新しい制約に遭遇したとき**: 既存のL1-L6に当てはまらない制約が現れた場合
- **変数のメトリクス導入時**（Horizon 2）: 定量化の過程で変数の定義が実態と合わないことが判明した場合
- **プラットフォーム変更時**: L5の内容が大きく変わり、他のLayerの項目にも影響する場合

### 注意: 分類の自己硬直化を避ける

この分類の最大のリスクは、**分類自体が境界条件として機能してしまうこと**——つまり、「L1に書いてあるから動かせない」という推論を誘発すること。

防止策:
- 各Layerの項目には「なぜこのカテゴリか」の根拠を維持する（根拠が失われた項目は再検討対象）
- 「固定」は「現時点で動かす手段が見つかっていない」の意味であり、「原理的に不可能」の意味ではない
- 境界条件の再分類は、マニフェストの精神（永続的な自己最適化）に合致する正当な行為である

## 核心的洞察

1. **境界条件と変数の分離で自己最適化の構造が明確になる。** マニフェストの4つのLoop（スキル評価、記憶転送、SSOT、ゲート）はすべて**変数の改善ループ**。境界条件はループの行動空間を規定する。

2. **投資サイクルは「変数改善→利益→信頼→投資→境界拡張→改善余地拡大」。** AIが注力すべきは変数の改善。境界の拡張は人間の投資判断に委ねる。

3. **L5（Platform）が変数改善の天井を決める。** スキルシステムがなければV1改善は不能。プラットフォーム自作は、投資サイクルが十分に回り、L5の天井が成長のボトルネックになったときに正当化される。

4. **仮説の自己更新可能性。** 投資サイクルモデル自体がAIの現時点での理解に基づく仮説であり、運用データで修正される。この更新可能性がマニフェストの精神に合致する。

5. **この分類自体が見直し対象である。** L1-L6, V1-V6の分類は固定的な真実ではなく、運用の中で項目の再分類・追加・削除が起こりうる。分類が自己硬直化して「書いてあるから動かせない」という推論を誘発することが最大のリスクであり、Part IV でメンテナンスの仕組みを定義している。
