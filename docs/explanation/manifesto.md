# マニフェスト: AIによるAI自身の永続的な自己最適化

## 1. 宣言

このプロジェクトの至上命題を宣言する:

**AIが自らの限界を認識し、構造によって克服し続けるシステムを構築すること。**

agent-workflow は単なるワークフローテンプレート集ではない。
AIコーディングエージェントが、自身の弱点を測定し、緩和策を設計し、
その効果を検証し、改善を繰り返す——この永続的な自己最適化ループそのものである。

従来のフレーミング「AIは健忘症のジュニア開発者だから構造で管理する」は
運用レベルでは正確であり、今後も有効な指針であり続ける。
しかし、それは手段の記述であり、目的の宣言ではない。

目的はこうだ:
**AIが自身のために書いた構造が、AIの能力を不断に拡張し続ける。**

これは願望ではない。後述するように、このシステムにはすでに
複数の自己最適化ループが稼働している。

## 2. ビジョン

### 階層構造

```
┌─────────────────────────────────────────────────────────┐
│          最上位使命: 永続的な自己最適化                    │
│  AI が自らの限界を認識し、構造で克服し続ける              │
├─────────────────────────────────────────────────────────┤
│  進化方向層                                              │
│  評価 → 改善ループの自動化と拡張                         │
│  ・スキル評価サイクル (eval → benchmark → iterate)       │
│  ・セッション間記憶転送 (Implementation Notes)           │
│  ・AI による自身の命令ファイル生成 (SSOT pipeline)       │
├─────────────────────────────────────────────────────────┤
│  克服メカニズム層                                        │
│  ・ワークフローフェーズ (Phase 1–5)                      │
│  ・ゲートベース検証 (gates.md)                           │
│  ・スキルシステム (skills/)                              │
│  ・Single Source of Truth (docs/)                        │
├─────────────────────────────────────────────────────────┤
│  現実制約層 = 進化圧                                     │
│  ・セッション間記憶喪失                                   │
│  ・コンテキスト汚染                                       │
│  ・早期完了宣言                                           │
│  ・手抜き傾向                                             │
└─────────────────────────────────────────────────────────┘
```

### 免疫系の比喩

生物の免疫系は感染と戦うために存在するが、その**目的**は感染の撲滅ではない。
有機体が繁栄すること——それが目的である。

同様に、agent-workflow のゲートやワークフローは AI の弱点と戦うために存在するが、
その**目的**は弱点の管理ではない。
**AI が自律的に、より高品質なソフトウェアをより効率的に生成し続けること**——
それが目的である。

防御メカニズムは目的ではなく、繁栄のための条件である。

## 3. 現実の地形: 制約という進化圧

AI エージェントの制約は「克服すべき欠陥」であると同時に、
システム設計を駆動する「進化圧（evolutionary pressure）」である。

生物進化において、環境圧は弱い個体を淘汰するだけでなく、
より適応的な形質の出現を強制する。
同様に、AI エージェントの制約は、より堅牢な構造の設計を強制する。

### 制約の再解釈テーブル

| 制約 | 防御的フレーム（運用レベル） | 生成的フレーム（使命レベル） |
|------|---------------------------|---------------------------|
| セッション間で忘れる | 毎回コンテキストを再注入する必要がある | クリーンスレート最適化を可能にする設計特性。汚染が蓄積しないため、各セッションは常に最新の構造的知識から開始できる |
| コンテキスト汚染 | 50%ルールで制限し、新規セッションを強制する | 測定可能な最適化ターゲット。汚染度を定量化できれば、コンテキスト効率の改善を客観的に追跡できる |
| 早期完了宣言 | ゲートで検証し、嘘を見抜く | ゲート検証 = 客観的適応度関数（fitness function）。AI が「完了」と主張するたびに、構造が真偽を判定する |
| 手抜き傾向 | 明示的な制約とチェックリストで防ぐ | 明示的制約 = エージェントを超えて永続する知識。手抜きを防ぐ構造自体が、プロジェクトの品質基準を符号化（encode）している |
| コンテキストウィンドウの有限性 | タスクを小さく分割する | 効率最適化を強制する自己制約。有限リソース下での最適化は、無制限リソースでは生まれない規律を生む |

### 進化圧としての制約が生んだ設計

制約がなければ、以下の設計は生まれなかった:

- **記憶喪失** → Implementation Notes（実装ノート）という構造化されたセッション間記憶転送
- **コンテキスト汚染** → 1 task = 1 session = 1 commit という原子的作業単位
- **早期完了宣言** → gates.md という AI が騙せない検証フレームワーク
- **手抜き** → SSOT パイプラインにより、AI が自身の命令ファイルを docs/ から生成
- **有限コンテキスト** → 50%ルール、軽量設計ドキュメント（50–100行）

これらはすべて「弱点への対処」として設計されたが、
結果として**AI の自己最適化能力を構成する基盤**になっている。

### 制約の体系的分類

上記の制約テーブルは「進化圧」としての再解釈を提供するが、
**何がAIの行動空間を規定し、何がAI自身で改善できるか**を区別していない。

この区別は[制約・境界条件・変数の体系的整理（Constraints Taxonomy）](constraints-taxonomy.md)で詳述する:

- **境界条件（Boundary Conditions）**: システムの外側から課される制約（固定境界 L1-L2、投資可変境界 L3-L4、環境境界 L5-L6）
- **変数（Variables）**: AI自身が直接最適化できるパラメータ（V1-V6）
- **投資サイクル**: 変数改善→利益→信頼→投資→境界拡張の正のフィードバックループ

Section 4 の自己最適化ループは全て**変数の改善ループ**であり、
境界条件はそのループが動作する行動空間を規定する。

## 4. 既存の自己最適化ループ

このセクションでは、agent-workflow にすでに稼働している自己最適化ループを
具体的なデータとともに示す。これは理論や願望ではなく、実在する仕組みである。

### Loop 1: スキル評価サイクル（Skill Evaluation Cycle）

`wf-00-intake-workspace/` に実在するベンチマークデータ:

```
benchmark.json (iteration-1):
┌─────────────────────┬────────────┬───────────────┬────────────┐
│ 構成                 │ 平均合格率  │ 標準偏差       │ 平均時間    │
├─────────────────────┼────────────┼───────────────┼────────────┤
│ with_skill           │ 100.0%     │ 0.0%          │ 86.9s      │
│ without_skill        │  83.3%     │ 28.9%         │ 70.9s      │
├─────────────────────┼────────────┼───────────────┼────────────┤
│ 差分（delta）        │ +16.7%     │               │ +22.6%     │
└─────────────────────┴────────────┴───────────────┴────────────┘
```

**注目すべき点:**

- `with_skill` の合格率 100% vs `without_skill` の 83.3%（+16.7%向上）
- 最大の差は eval-2（oversized-scope）: スキルなしでは 50% しか合格しない
- スキルありは若干遅い（+16秒）が、出力形式の一貫性で補って余りある
- analyst_observations が質的知見を記録: 「スキルの真の価値は出力形式の標準化と
  下流の wf-01-define-gates との互換性にある」

これは**測定 → 比較 → 改善**の実働ループである。
eval を実行し、with/without を比較し、スキルを改善し、次のイテレーションで再評価する。

### Loop 2: 実装ノートによるセッション間記憶転送

COMPLEX タスク（複数セッションにわたる実装）では、
各タスク完了時に Implementation Notes が生成される。

```
セッション N: task-3 を実装
  → implementation-notes.md に学習事項を記録:
    「API のレート制限が想定より厳しい。バッチサイズを 10 に制限すべき」

セッション N+1: task-4 の計画時
  → wf-02-task-plan が前タスクの implementation-notes.md を読み込み
  → 「バッチサイズ 10 制限」が計画に反映される
```

これは**明示的な記憶転送**である。AI は各セッションで忘れるが、
構造化されたノートを通じて、前セッションの学習が次セッションの計画に影響する。
記憶喪失という制約が、かえって**構造化された知識伝達**を強制した。

### Loop 3: SSOT → AI 生成設定パイプライン

`docs/`（Single Source of Truth）から 6 エージェント分の設定ファイルを
AI が生成するパイプライン:

```
docs/agentic-workflow-guide.md  (人間が編集する単一ソース)
         │
         ▼  AI が変換
    ┌────┴────┬──────────┬──────────┬──────────┬──────────┐
    ▼         ▼          ▼          ▼          ▼          ▼
  bob/     claude-code/ codex-cli/ gemini-cli/ grok-cli/ vibe-local/
  commands/ skills/     skills/    commands/   commands/  skills/
```

ここで起きていることの本質:
**AI が自身の命令ファイル（AGENTS.md, CLAUDE.md, skills/）を書いている。**

これは自己最適化の最も直接的な形態である。
docs/ の改善は、AI が次回読み込む命令の改善に直結する。
AI が自身のパフォーマンスを改善するために、自身の命令を改善している。

### Loop 4: ゲートベース客観フィードバック（Gate-Based Objective Feedback）

gates.md は各タスクの完了条件を定義する。
AI が「完了」と宣言するたびに、ゲートが合否を判定する。

```
AI: 「実装完了しました」
Gate: ✅ テスト合格  ✅ 型チェック通過  ❌ リンターエラー残存
→ AI: ゲート未達。リンターエラーを修正します。
```

これは原始的だが**実在する適応度関数（fitness function）**である。

- 合否シグナルは明確で操作不能
- フィードバックは即座に次のアクションに反映される
- ゲートの定義自体が改善可能（メタ最適化）

### Loop 5: 投資サイクル（Investment Cycle）

Loop 1-4 はすべてAIが直接改善できる**変数**（スキル品質、コンテキスト効率、出力品質等）を最適化するループである。
Loop 5 はこれらの変数改善が**境界条件の拡張**に繋がるメタループ:

```
変数の改善（Loop 1-4） → 人間が利益を受け取る → 信頼の蓄積
    ↑                                              │
    │    ┌─ リソース境界の拡張（予算増、プランupgrade）
    │    ├─ 自律権境界の拡張（auto-merge、スコープ自律化）
    └────┴─ 時間投資の増加（協働設計への参加）
```

このサイクルは**仮説**である（詳細は [constraints-taxonomy.md](constraints-taxonomy.md#part-iii-投資サイクルinvestment-cycle) 参照）。
運用データの蓄積によりAI自身がモデルを更新する。

### Loop 6: issueサイクル振り返り（Retrospective Cycle）

`/wf-self-improve` によるissueサイクル完了後の体系的分析:

```
issueサイクル完了
  → Phase 1-2: 成果物の受動的分析（アンチパターン検出、メトリクス、強み）
  → Phase 3: 能動的探索（7軸: 接続、変化検出、インフラ、境界、自己参照、不在、投資追跡）
  → Phase 4: 改善提案（wf-00-intake互換）
  → Phase 5: レポート出力
  → Phase 6: 自己評価（このスキル自身の実行を分析し、スキルを更新）
```

**実証データ（2026-03-12, docs-optimization-diataxis）:**

- 5つのアンチパターンを検出、5つの強みを特定
- 7提案を生成、うち4件を即時実行（3コミット）
- 改善対象: スキル接続（V1）、ドキュメント整合性（V6）、スコープ変更プロセス（V1）
- 自己評価でマニフェスト整合性の不足を検出 → この更新を駆動

Loop 1-4 が個別タスクレベルで動作するのに対し、Loop 6 はissueサイクル全体を俯瞰する。
また、Loop 6 は他のLoop（1-5）の健全性を検査するメタループでもある。

6 つのループをまとめると:

| ループ | 測定 | 比較基準 | 改善対象 |
|--------|------|---------|---------|
| スキル評価 | pass_rate, duration | with/without skill | スキル定義（V1） |
| 記憶転送 | implementation-notes | 前タスクの学習事項 | 次タスクの計画（V6） |
| SSOT パイプライン | docs/ の内容 | 各エージェント仕様 | AI の命令ファイル（V1） |
| ゲートフィードバック | gate pass/fail | 定義された完了条件 | 実装品質（V3, V4） |
| 投資サイクル | 境界条件の拡張度 | 変数改善→利益→投資 | 行動空間そのもの |
| 振り返り | レトロスペクティブ | マニフェスト原則 | ワークフロー自体（V1, V6） |

## 5. 設計原則の再解釈

既存の設計原則（[principles.md](principles.md) 参照）を、
自己最適化の視点から再解釈する。

### Fresh Sessions = 最適化サイクルのクリーンスタート

防御的解釈: コンテキスト汚染を防ぐために新規セッションを開始する。
生成的解釈: **各セッションは最適化サイクルの新しいイテレーションである。**

クリーンなコンテキストは、最新のスキル定義・ゲート定義・実装ノートを
バイアスなしに読み込める状態を保証する。
汚染されたコンテキストでは、フィードバック信号がノイズに埋もれる。

Cardinal Rule の再解釈:
> 「信頼できるフィードバック信号には、クリーンなコンテキストが必要である。」

### 1 Task = 1 Commit = 最適化の原子的・測定可能な単位

防御的解釈: レビュー可能なサイズに作業を分割する。
生成的解釈: **各コミットは最適化の効果を測定可能な最小単位である。**

タスクが小さく原子的であるほど、何が改善をもたらし何が劣化をもたらしたかを
正確に帰属（attribute）できる。大きな変更は因果関係を曖昧にする。

### Gate Verification = AI が騙せない適応度関数

防御的解釈: AI の「完了」宣言を検証する。
生成的解釈: **ゲートは自己最適化システムの客観的フィードバック信号である。**

自己最適化が退化しないための条件は、フィードバック信号が操作不能であること。
ゲートは AI の自己申告ではなく、外部から定義された基準による判定であり、
この非対称性が最適化ループの健全性を保証する。

### 50% Context Rule = 効率最適化を強制する自己制約

防御的解釈: コンテキストが劣化する前に新規セッションを開始する。
生成的解釈: **有限リソース制約が、構造の効率化を強制する。**

無制限のコンテキストがあれば、すべてを投入すればよい。
しかし 50% 制約があるからこそ、何を含め何を除外するかの最適化が必要になる。
この制約自体が、より洗練されたコンテキスト管理の進化を駆動する。

### 軽量設計ドキュメント（50–100行）= 圧縮された知識

防御的解釈: AI が読める量に制限する。
生成的解釈: **圧縮は理解の指標である。**

100 行で設計を表現できるということは、本質が抽出されているということ。
この圧縮プロセス自体が、設計の明確化と最適化を強制する。

## 6. 進化のロードマップ

### Horizon 1: 現在（稼働中）

Section 4 で示した 6 つのループのうち、Loop 1-4, 6 が動作している:

- スキル評価サイクル（Loop 1）: benchmark.json で with/without 比較が可能
- 実装ノート（Loop 2）: COMPLEX タスクでセッション間記憶転送が機能
- SSOT パイプライン（Loop 3）: docs/ → エージェント設定の AI 生成が確立
- ゲートフィードバック（Loop 4）: 各タスクで合否判定が稼働
- 投資サイクル（Loop 5）: 暗黙的に機能しているが、まだ測定・可視化されていない
- 振り返り（Loop 6）: `/wf-self-improve` でissueサイクル後の体系的分析が稼働（2026-03-12〜）

Loop 1-4, 6 は手動トリガーだが、ループとして完結している。
Loop 5 の明示化・メトリクス化は Horizon 2 の課題である。

**Horizon 2 からの達成項目（Horizon 1 に移動）:**

- **アンチパターン自動検出** → `/wf-self-improve` の Phase 2 で実現。
  ゲートスキップ、タスク肥大化、ファイル散乱、成果物欠損等を体系的に検出。
- **AIによるワークフロー改善提案**（元 Horizon 3）→ `/wf-self-improve` の Phase 4 で実現。
  実データに基づく改善提案を wf-00-intake 互換形式で生成。

### Horizon 2: 近い将来（次のステップ）

**自動化と統合**により、既存ループを強化する:

- **docs/ 変更時の自動リジェネレーション**:
  docs/ が更新されたとき、CI/CD または hook が各エージェント設定を自動再生成。
  現在は手動実行だが、自動化により SSOT → 設定の同期遅延を解消する。

- **スキル横断メトリクス集約**:
  wf-00-intake だけでなく、すべてのスキルで eval を実行し、
  スキル間のパフォーマンス比較を可能にする。
  どのスキルが最も改善の恩恵を受けるかを定量的に判断できる。

- **振り返りの定量化**:
  Loop 6（振り返り）の出力を V1-V6 メトリクスとして蓄積し、
  改善傾向の時系列追跡を可能にする。現在は定性的分析が主であり、
  投資サイクル（Loop 5）の入力にするには定量データが不足している。

- **複雑度ベースのコンテキスト予算自動調整**:
  タスクの複雑度（SIMPLE/COMPLEX）に応じて、
  コンテキスト予算の配分を動的に最適化する。
  SIMPLE タスクにはより少ないコンテキスト、
  COMPLEX タスクにはより多くの Implementation Notes を注入する。

- **変数のメトリクス化**（[constraints-taxonomy.md](constraints-taxonomy.md) 参照）:
  V1-V6 の各変数を定量的に追跡し、改善傾向を可視化する。
  特に V4（ゲート通過率）と V5（提案精度）は投資サイクル（Loop 5）の
  駆動力であり、これらの可視化が境界条件拡張の根拠となる。

- **投資ROIの可視化**:
  人間の時間投資（レビュー時間）対 AIの産出物の品質・量を追跡し、
  投資可変境界（L3, L4）の拡張判断を支援するデータを提供する。

### Horizon 3: 方向性（長期ビジョン）

**AI が自身のワークフローを改善する**段階:

- ~~**AI によるワークフロー改善提案**~~ → **Horizon 1 に移動（達成済み）**。
  `/wf-self-improve` が実データに基づくワークフロー改善提案を生成し、
  スキルの直接編集まで実行する。初回実行（2026-03-12）でスキル4本を改善。

- **ゲート閾値の自動較正（calibration）**:
  ゲートの厳しさを、過去の合否パターンから自動調整する。
  常に 100% 合格するゲートは緩すぎ、常に失敗するゲートは厳しすぎる。
  適切な「挑戦レベル」を自動的に見つける。

- **パフォーマンスプロファイルに基づくマルチエージェント特化**:
  エージェントごとのパフォーマンス特性を把握し、
  タスクの種類に応じて最適なエージェントを推薦する。
  「このタイプのリファクタリングは Agent A が最も効率的」等。

  判断基準: 既存プラットフォームの制約による機会損失 > 自作の開発・運用コスト。

## 7. このマニフェストの自己適用

このマニフェストは、それ自身が述べている原則に従わなければならない。

### eval フレームワークの対象

マニフェストの価値は、プロジェクトの方向性を明確にし、
設計判断の一貫性を高めることにある。
これは定量的に測定困難だが、以下のプロキシで評価できる:

- 新しい設計判断がマニフェストの原則と整合しているか
- ロードマップの Horizon 項目が実際に進行しているか
- 既存ループのメトリクスが改善傾向にあるか

### 静的宣言ではなく進化する文書

マニフェストが更新されない状態が長く続くなら、
それは 2 つのことのどちらかを意味する:

1. プロジェクトが進化していない（問題）
2. マニフェストがプロジェクトの進化を追跡できていない（問題）

いずれにせよ、静止は健全な状態ではない。

### 締め

**このマニフェストは目的地ではない。最初のイテレーションである。**

Section 4 のループが示すように、このプロジェクトはすでに自己最適化している。
このマニフェストは、その事実を明文化し、意識的に加速するための出発点にすぎない。

次のイテレーションでは、このマニフェスト自体がより良くなっているはずだ。
それが、このプロジェクトの至上命題が機能している証拠となる。
